# Abstract

This specification introduces a new approach and the tooling for building (packing), running integration tests and publishing for charm repositories. The new approach is based on a flexible, local-first model based on a build plan, a build output, and spread-based test execution.

# Rationale

In the Platform Engineering group, [operator-workflows](https://github.com/canonical/operator-workflows) is the repository holding the shared workflows for most of the charms repositories. It is used by around 50 repositories in the Canonical organization. Thanks to the shared workflows, we ensure consistency, simplify maintenance and reduce boilerplate. Most of the shared workflows are straightforward with the main exception of the [integration\_test.yaml](https://github.com/canonical/operator-workflows/blob/main/.github/workflows/integration_test.yaml) workflow, the main concern of this specification.

[integration\_test.yaml](https://github.com/canonical/operator-workflows/blob/main/.github/workflows/integration_test.yaml) has served us well, as a shared workflow to build artifacts and run integration tests. However, it is showing its limitations:

- Lack of support for monorepos. It is not capable of associating charms and rocks and the publish workflow is not prepared for monorepos. Also it does not provide easy tooling to support building charms and rocks sharing code until \*craft supports it natively.  
- Lack of support for combinations of bases, architectures, and test matrices in a single workflow. Publishing across different bases and architectures within the same workflow is equally inflexible.  
- Scope too wide in integration tests. As of Mar 24, 2026 [integration\_test.yaml](https://github.com/canonical/operator-workflows/blob/main/.github/workflows/integration_test.yaml) has 50 input variables and lots of mixed concerns, like zap, load tests, trivy, provisioning, devstack installation… It has organically grown as a mixture of yaml/typescript/bash, which is hard to understand and maintain. Adding or removing extra functionality is prone to making many repositories fail.  
- It is not trivial to replicate what the workflow does in a local environment. Pipelines are not self documented.  
- Lack of pinning. This increases the blast radius of any change and increases the risk of security incidents like supply chain attacks.  
- High cognitive complexity. It is difficult to reason about and, subsequently, debug the behavior of the pipeline.  
- Lack of extensibility. Scripts at certain points are starting to appear as “poor man's extension points”. Can’t be executed outside of the GitHub platform, causing platform lock-in and longer feedback development cycles.  
- Can’t be unit tested.

Several ideas from other teams' shared workflows have been taken into consideration for this specification, but none of the solutions investigated fulfilled the objectives set in this specification.

# Specification

## Scope of the specification

Building, integration testing and publishing charms and its associated rocks and snaps.

Specifically, redesign of the [integration\_test.yaml](https://github.com/canonical/operator-workflows/blob/main/.github/workflows/integration_test.yaml) and  [publish](https://github.com/canonical/operator-workflows/blob/main/.github/workflows/publish_charm.yaml) workflows. 

- Building the artifacts: charms, rocks and snaps.  
- Provision the environment to deploy the artifacts and run tests.  
- Running integration and end-to-end tests.  
- Publishing artifacts to the stores from a validated run.

## Objectives for operator-workflows redesign

The proposal fulfills  the following non-functional requirements:

- Flexibility. Support any repository layout — monorepos, 12-factor charms, non-standard structures. Enable combinatorial testing across architectures, Juju versions, and substrates. Allow teams to define custom provisioning and test execution logic without forking the tool.  
- Modularity. Provide extension points so the pipeline is flexible and does not grow without bounds. At minimum: external actions or pipelines can consume build outputs without coupling to the tool itself.  
- Local testing. Make it straightforward to build artifacts, provision the environment, and run tests locally, with behavior matching the CI pipeline as closely as possible.  
- Performance. The new design must not be slower than the current pipeline. Provisioning should run in parallel to building artifacts and all optimizations for caching artifacts should be retained.  
- Debuggability. Make it easy to inspect and reproduce failing jobs.  
- Convention over configuration. Support flexible repository layouts, with sensible defaults for standard ones.  
- Reproducibility. The full pipeline and its dependencies must be pinned.  
- Easy migration. Existing repositories should be able to adopt the new pipeline incrementally.

## Overview

### **How integration\_test.yaml works today**

Currently, [integration\_test.yaml](https://github.com/canonical/operator-workflows/blob/main/.github/workflows/integration_test.yaml) is monolithic but internally has three phases:

-  The first part is the “[plan](https://github.com/canonical/operator-workflows/blob/e243dac0b57ff685d35033fe1003335168bd9910/.github/workflows/integration_test.yaml#L164)”. The plan looks for the artifacts (charms, rocks, docker and other resources) in the repository and generates a plan with the list of the artifacts and the output location for the build.   
- The “[build](https://github.com/canonical/operator-workflows/blob/e243dac0b57ff685d35033fe1003335168bd9910/.github/workflows/integration_test.yaml#L201)” jobs run in a matrix (one per artifact identified in the plan job).   
- The “[integration-test](https://github.com/canonical/operator-workflows/blob/e243dac0b57ff685d35033fe1003335168bd9910/.github/workflows/integration_test.yaml#L362)” job runs after the “plan” in a matrix for each module to test. It runs many steps, that can be somewhat grouped into:  
  - Provisioning the machine, tooling…   
    - There are some GitHub specific configurations. Configuring the credentials for the [ghcr.io](http://ghcr.io) registry and using a [docker.io](http://docker.io) mirror in self-hosted runners.  
  - Plan-provisioning step, that waits for the build jobs to finish, gets the artifacts and prepares the arguments for the integration tests with the list of artifacts.  
  - Run the integration test.

The integration\_test.yaml receives all its inputs through input variables. None of its parts can be run locally.

### **What changes in the new design**

The new design preserves the same three phases (plan, build, test) but makes each one explicit and composable. A new CLI tool called opcli ties these pieces together — generating configuration files, building artifacts, running concierge, expanding and running spread, and assembling pytest arguments — so each phase can be executed  both locally and in CI with a single command. Four key changes enable this.

**Explicit build plan (artifacts.yaml)**. The list of artifacts — charms, rocks, and the relations between them (e.g. which rocks are resources of which charms) — is declared in an artifacts.yaml  file. This makes the build plan inspectable and editable, rather than inferred at runtime. For standard repository layouts,  “opcli artifacts init”  can generate the file automatically. It also supports building multiple charms or rocks from the same source directory — something charmcraft and rockcraft don't natively allow — by using alternate configuration filenames that the tool resolves transparently at build time.

**Build output as a stable interface (artifacts.build.yaml).** After the  artifacts are built (locally or in CI), the build step produces an artifacts.build.yaml  file that extends the build plan with the paths of the built artifacts. This file is the explicit input to the test  phase, and can also be consumed by other pipelines, like publishing, Trivy scanning or load testing, without coupling them to the integration-test workflow.

**Concierge for environment provisioning**. [Concierge](https://github.com/canonical/concierge/)  replaces the ad-hoc provisioning steps scattered across the current workflow. A  concierge.yaml  file in the repository declaratively describes the test environment (LXD, MicroK8s, Juju controller, etc.). Concierge serves a dual purpose: it is a simple provisioning tool that enables local testing, and it is the bridge between local and CI environments — differences such as image mirrors are expressed  as overrides to the same concierge file.

**Spread for test execution**. [Spread](https://github.com/canonical/spread)  orchestrates the full test run (provisioning \+ test execution) through a spread.yaml file. The user declares a single logical backend (integration-test) in spread.yaml and opcli expands it into the appropriate concrete backend depending on whether the test runs locally or in CI:

- Local: provisions an LXD VM, runs concierge inside it, and uploads OCI images to a local registry.  
- CI: runs on the current machine, uses a CI-adapted concierge file, waits for artifacts built in parallel, and downloads them.

After backend provisioning, spread jobs run the integration tests. For standard repositories,  spread.yaml  and  task.yaml  can be auto-generated with test auto-discovery. For more complex setups  (multiple Juju versions, different Kubernetes substrates, extra provisioning), the user customizes the spread files directly. Spread's prepare  and  prepare-each  hooks provide additional extension points beyond what concierge covers. 

## New workflows

### **CI (GitHub)**

The CI workflow mirrors the local sequence but distributes work across  parallel GitHub jobs: 

1. **Plan**. Reads  artifacts.yaml  and produces a build matrix (one entry per  artifact).  
2. **Build (matrix).** Each job builds one artifact (charm, rock, etc.) and uploads a per-arch partial build manifest (``artifacts-build-{type}-{name}-{arch}``) as a GitHub Actions artifact.  
3. **Test plan**. Uses  spread.yaml to run \`opcli spread jobs\`  to discover the test jobs and produces a test matrix (one entry per test execution).  
4. **Test (matrix).** Each job runs a single spread test with the  ci:  backend, which targets the current GitHub runner. Inside each job:  
   1. Backend prepare  
      1. opcli is installed  
      2. concierge.yaml  is patched with docker mirrors and any other CI specific overrides.  
      3. The machine is provisioned with Concierge.  
      4. The job waits for the per-arch partial build manifests to appear, downloads and merges them locally (``opcli artifacts fetch --wait``), then downloads all built artifact archives.  
   2. The integration test is executed following a normal spread job.

### **Local running with spread**

Very similar to the GitHub workflow, spread can be run locally. It uses a LXD virtual machine as the backend:

```shell
opcli artifacts init
opcli artifacts build

# creates spread.yaml and tests/run/task.yaml.
opcli spread init
# "opcli spread run" is a wrapper for spread that expands the backends and runs spread as a subprocess
opcli spread run -list
opcli spread run local:ubuntu-26.04:tests/integration/run:test_charm
```

### **Local running without spread**

For fast feedback, spread can sometimes be inconvenient. As an alternative, the steps can be run manually, similarly to how the provision in the backend is expanded:

```shell
opcli artifacts init
opcli artifacts build

# Runs `sudo concierge prepare -c concierge.yaml`
opcli env provision -c concierge.yaml
# Creates a k8s registry if needed and loads the OCI images
opcli env deploy-registry
opcli artifacts push-images

# If spread.yaml or task.yaml was modified manually, the specific provisioning scripts should be executed here.
# Runs "tox -e integration -- --charm-file=... ..."
opcli pytest run -- -k test_charm
```

## Contract for the repository and pipeline steps

The following files serve as the interfaces for the different steps of the pipeline. Some of them can reside in the repository.

| File | Required in repository | Purpose |
| :---- | :---- | :---- |
| artifacts.yaml | Optional. Can be autogenerated or created manually. | Declares the charms, rocks, images, snaps in the repository. It also links charms with its resources. |
| artifacts.build.yaml | No. Generated by the build step. Should be in .gitignore. | Extends the artifacts.yaml with the paths or references of the build artifacts. Consumed by the test and publish steps. |
| concierge.yaml | Yes. | Declarative provisioning \- juju, microk8s, ck8s, lxd, addons, constraints, host packages… |
| spread.yaml  | Optional. Can be autogenerated or created manually. | Spread configuration: The backend section is expanded by opcli to adapt to local/GitHub and provision the machine It defines the jobs for all the integration tests. |
| tests/integration/run/task.yaml | Optional. Can be autogenerated or created manually. | Spread task with information on how to run an integration test. |

### **artifacts.yaml**

```
version: 1
rocks:
  - name: indico
    rockcraft-yaml: indico_rock/rockcraft.yaml
    platforms:
      - arch: amd64
        runner:
          - ubuntu-22.04
      - arch: arm64
        runner:
          - self-hosted
          - ubuntu-24.04-arm
  - name: indico-nginx
    rockcraft-yaml: nginx_rock/rockcraft.yaml
charms:
  - name: indico
    charmcraft-yaml: charmcraft.yaml
    channel: latest/edge          # optional: per-charm publish channel
    resources:
      indico-image:
        type: oci-image
        rock: indico
      indico-nginx-image:
        type: oci-image
        rock: indico-nginx
  - name: indico-machine
    charmcraft-yaml: machine-charm/charmcraft.yaml
    channel: 1.0/stable           # optional: different channel for this charm
```

The optional `channel` field on a charm entry sets the default CharmHub channel used by `opcli artifacts publish`. It takes precedence over the global `--channel` flag. If neither is set for a charm, `opcli artifacts publish` raises an error.

### **spread.yaml**

The  spread.yaml  file defines virtual backends that  opcli  expands into the real  local:  or  ci:  backends  before running spread. The primary virtual backend type is  `integration-test`, used for integration tests with Juju/concierge provisioning. A lightweight `opcli-minimal` type is also available for scenarios that only need opcli itself (e.g., tutorial runs) without concierge or Juju. This file can be auto-generated with "opcli spread init"  (which discovers all integration tests in the repository) and then optionally edited and committed. Similar to  "charmcraft test"  and  "snapcraft test", the backend expansion happens transparently using "opcli spread run".

The virtual backend supports opcli-only keys (`runner`, `cpu`, `memory`, `disk`) that are stripped during expansion. Pytest invocation is customized per-suite via Jinja2 templates (`pytest-arguments-template`, `pytest-environment-template`) — see the `integration-suites` section below.

The following is a modified version of "opcli spread init", where there are several systems for different architectures and variants for different Concierge configurations:

```
project: indico-operator                     
path: /home/ubuntu/proj  
kill-timeout: 60m             
warn-timeout: 1m                                      
backends:               
  integration-test:                                                                 
    type: integration-test          
    systems:       
      - ubuntu-24.04:                 
          runner: [self-hosted, noble]
      - ubuntu-24.04-arm64:                    
          runner: [self-hosted, arm64]
          arch: arm64
environment:
  CONCIERGE: '$(HOST: echo "${CONCIERGE:-concierge.yaml}")'
  CONCIERGE/test_indico_ck8s: concierge-ck8s.yaml
  OPCLI_GIT_REF: '$(HOST: echo "${OPCLI_GIT_REF:-main}")'
suites:
  tests/integration/:
    summary: integration tests
    backends:
      - integration-test
    environment:
      MODULE/test_indico: test_indico.py
      MODULE/test_indico_ck8s: test_indico.py
      MODULE/test_indico_machine: test_indico_machine.py

```

“opcli spread expand “ can be used to preview the fully expanded file without running it.

Users can customize the full  spread.yaml  (adding backends, suites, environment variables, etc.), but if the changes diverge significantly from the conventions, bypassing spread with  “opcli env provision”  \+  “opcli pytest run”  may no longer replicate the same behavior.

### **task.yaml**

`task.yaml` files are **generated at expand/run time** (not by `spread init`) into `build/<suite>/run/task.yaml`. The generated content looks like:

```
summary: integration tests

execute: |
    cd "${SPREAD_PATH}"
    PYTEST_CMD=$(opcli pytest expand -e "${TOX_ENV:-integration}" --module "${MODULE}") || exit 1
    OPCLI_PACKAGE='${OPCLI_PACKAGE:-opcli}'
    runuser -l ubuntu -s /bin/bash -c "
        cd '${SPREAD_PATH}' && \
        OPCLI_PACKAGE='${OPCLI_PACKAGE}' \
        OPCLI_ARTIFACTS_BUILD_YAML='${OPCLI_ARTIFACTS_BUILD_YAML}' \
        ${PYTEST_CMD}"
```

The  MODULE  environment variable is set by the spread variant (defined in  spread.yaml ). Each variant runs one test module. MODULE values are **relative file paths from `OPCLI_CWD`** (the directory pytest is invoked from) — e.g. `tests/integration/test_deploy.py` for a suite at `tests/integration/` with `working-dir: ./`. This ensures pytest can resolve the path from its working directory. MODULE keys use the stem with path separators replaced by underscores (e.g. `MODULE/test_deploy`, `MODULE/subdir_test_upgrade`) — dots are not valid in spread environment variable names. Auto-discovery is **recursive** — test files in subdirectories are found automatically. Since spread executes scripts as root, `opcli pytest expand` assembles the command and `runuser` drops privileges to ubuntu before running tox/pytest (which needs the user's Juju credentials and kubeconfig).

`opcli pytest expand` reads per-suite template configuration from `spread.yaml` to assemble the full pytest invocation. Artifact fixtures are injected automatically by the `pytest-opcli` plugin — no CLI flags needed for that. To pass Juju-specific options (e.g. `--model`, `--keep-models`) or any other pytest flags, use `pytest-arguments-template` on the suite entry in `spread.yaml`:

```yaml
integration-suites:
  tests/integration/:
    pytest-arguments-template: |
      --model testing
      --keep-models
```

Templates have access to: `artifacts` (full `ArtifactsGenerated` model), `arch` (current architecture string), and `env` (snapshot of the current process environment). `env` works in **both** `pytest-arguments-template` and `pytest-environment-template`. Use it to inject dynamic values — for example reading a Juju model name from a spread env var:

```yaml
integration-suites:
  tests/integration/:
    pytest-arguments-template: |
      --model={{ env.get("JUJU_MODEL", "testing") }}
      --keep-models
```

Use `pytest-environment-template` to forward environment variables from the root spread shell into tox — for example to pass `SPREAD_JOB` or CI tokens:

```yaml
integration-suites:
  tests/integration/:
    pytest-environment-template: |
      SPREAD_JOB={{ env.get("SPREAD_JOB", "") }}
      GITHUB_TOKEN={{ env.get("GITHUB_TOKEN", "") }}
```

Because `opcli pytest expand` runs as root in the spread task, its process environment contains root's variables. The rendered `KEY=value` pairs are baked into the `$PYTEST_CMD` string that is then passed to `runuser`, making them available to tox.

> **Important:** variables forwarded via `pytest-environment-template` only reach pytest if tox also passes them through. Add the variable names to `passenv` in your `tox.ini`:
> ```ini
> [testenv:integration]
> passenv =
>     SPREAD_JOB
>     GITHUB_TOKEN
> ```

**`env` access patterns:** `env.get("VAR", "")` returns a default when the variable is absent (recommended). `env.VAR` and `env["VAR"]` raise `ConfigurationError` if the key is missing — use these as a self-documenting "this must be set" assertion.

This makes `task.yaml` universal — the same generated file works for every project regardless of its conftest.py conventions.

## Tooling (opcli)

The CLI tool opcli  facilitates auto-generation of configuration files, building artifacts, provisioning and running tests. It abstracts the differences between local and CI execution.

The functionality for opcli is grouped into four command families:  artifacts , environment,  spread  and  pytest . Additional commands and subcommands can  be added as needed.

### **opcli artifacts**

| Command | Description | Extra options |
| :---- | :---- | :---- |
| opcli artifacts init | Discovers charms, rocks and snaps in the repository and generates artifacts.yaml | –force. Overwrites existing files. |
| opcli artifacts build | Builds all artifacts declared in artifacts.yaml and produces artifacts.build.yaml If charms or rocks arg is used, only those artifacts will be built and artifacts.build.yaml updated (or created only with some build artifacts). | –charm \<charm-name\>, charms to build. Can be used several times. –rock \<rock-name\>, rocks to build. Can be used several times. –snap \<snap-name\>, snaps to build. Can be used several times. --build-timeout \<seconds\>: maximum seconds for each pack invocation (charmcraft/rockcraft/snapcraft). Defaults to 3600 (1 hour). |
| opcli artifacts fetch | Downloads per-arch partial build manifests from a CI run, merges them locally, downloads all built artifact archives, then rewrites local paths so `opcli pytest run` and `opcli spread run` work without a local build. Requires `artifacts.yaml` to be present in the working directory. | --run-id \<id\>: GitHub Actions run ID (required). --repo \<owner/name\>: repository, inferred from git remote if omitted. --arch \<arch\|all\>: architecture to fetch; defaults to the current machine's arch (auto-detected). Use `all` to fetch every architecture (e.g. for publish workflows). --wait: retry until all expected partial manifests appear (use when the build job may still be running). --wait-timeout \<seconds\>: maximum seconds to wait; implies --wait. Defaults to 1800 (30 min). |
| opcli artifacts push-images | Loads the image artifacts (rocks) into the local image registry. | \-r registry. Specify the image registry. Defaults to localhost:32000 |
| opcli artifacts path | Prints absolute path(s) to built artifacts from artifacts.build.yaml. One path per line. | Optional positional NAME to filter by artifact name. --type (charm\|rock\|snap). --arch to override architecture. |
| opcli artifacts publish | Uploads charms and their OCI-image resources to CharmHub. | --channel \<channel\>: global default channel (e.g. latest/edge, 1.0/stable). Optional when every charm declares a `channel` in artifacts.yaml. Per-charm channel takes precedence. --charm \<name\>: publish only this charm (repeatable). --dry-run: show what would be published. --json: structured JSON output. |

runuser -l ubuntu -s /bin/bash -c 'set -ex; . <(opcli tutorial expand -- "$1")' _ "${SPREAD_PATH}${TUTORIAL}"
```

**Markdown extraction:** 3-backtick code fences are included (except language tags starting with `{`, e.g. `` {terminal} ``). 4+-backtick fences are excluded. Blocks opened by `<!-- SPREAD` and closed by `-->` (a single HTML comment) are always included. Use `<!-- SPREAD SKIP -->` / `<!-- SPREAD SKIP END -->` to exclude regions entirely (e.g. local-only setup steps). Note: there is no `<!-- SPREAD END -->` in Markdown — the closing delimiter is simply `-->`.

**RST extraction:** `.. code-block::` directives (directive options like `:caption:` are skipped; internal blank lines within the code body are preserved). `.. SPREAD` / `.. SPREAD END` blocks. `.. SPREAD SKIP` / `.. SPREAD SKIP END` ranges.



## New shared workflows

### **build\_artifacts.yaml**

From the artifacts.yaml it generates all the built artifacts and the artifacts.build.yaml file.

It will have three jobs, each one depending on the previous:

- build-matrix. It will produce the build matrix for the building of all artifacts.  
- build: matrix that will build every artifact in parallel. Each job uploads a per-arch partial build manifest (``artifacts-build-{type}-{name}-{arch}``) as a GitHub Actions artifact. There is no separate collection step — ``opcli artifacts fetch`` merges the partials locally at fetch time.  

**Workflow inputs:**

| Input | Type | Default | Description |
| :---- | :---- | :---- | :---- |
| `working-directory` | string | `.` | Directory containing artifacts.yaml |
| `upload-image` | string | `""` | Rock upload mode (`registry` or `artifact`). Auto-detected if empty. |
| `pre-build-rock-script` | string | `""` | Shell command to run before each rock build. Skipped on cache hits. |
| `build-timeout-minutes` | number | `60` | Maximum minutes for each build job. Also passed as `--build-timeout` to `opcli artifacts build` so the subprocess timeout tracks the job timeout. |

	

### **integration\_tests.yaml**

The integration tests will run two jobs in parallel, the first one being the build\_artifacts.yaml (or equivalent).

In parallel to the building of artifacts it will run the following jobs, each one depending on the previous:

- test-matrix: produces the spread job matrix (opcli spread jobs / spread \-list). This completes quickly and immediately unblocks the test jobs.  
- test: runs each spread job in a matrix. The test job depends only on test-matrix, so it starts provisioning (concierge, LXD/MicroK8s, etc.) in parallel with the artifact build.  Artifacts are downloaded inside the spread CI backend prepare step, at the point they are actually needed — after provisioning is complete.

A detached tmate session can be included at the beginning of the build and test jobs to facilitate debugging. It is activated only when the run is in debug mode (runner.debug \== '1'); self-hosted runners use `canonical/action-tmate` and GitHub-hosted runners use `mxschmitt/action-tmate`.

**Workflow inputs:**

| Input | Type | Default | Description |
| :---- | :---- | :---- | :---- |
| `working-directory` | string | `.` | Directory containing artifacts.yaml and spread.yaml |
| `upload-image` | string | `""` | Rock upload mode passed to build-artifacts |
| `pre-build-rock-script` | string | `""` | Shell command to run before each rock build |
| `build-timeout-minutes` | number | `60` | Maximum minutes for each artifact build job. Passed to build-artifacts. |
| `test-timeout-minutes` | number | `120` | Maximum minutes for each spread test job. |
| `spread-jobs-include` | string | `""` | fnmatch pattern to restrict the spread job matrix |
| `test-secret-1-name` … `test-secret-5-name` | string | `""` | Names of secrets to pass to integration tests |

### **doc\_test.yaml**

A lightweight reusable workflow for documentation and tutorial tests. Unlike `integration_tests.yaml`, it has no artifact build step — the test environment only needs uv and opcli installed (via the `opcli-minimal` virtual backend).

It runs two sequential jobs:

- test-matrix: produces the spread job matrix (`opcli spread jobs [--include ...]`). Accepts a `spread-jobs-include` fnmatch pattern to restrict the matrix (e.g. `docs-ci:*`).
- test: runs each spread job in a matrix. Jobs start as soon as the matrix is available.

The workflow pins the `opcli` version via `canonical/get-workflow-version-action`, matching the behaviour of the other shared workflows.

### **opcli artifacts fetch — known limitations**

The following limitations are inherent to the "no collect job" design and should
be understood when diagnosing CI failures:

1. **Auto-detect arch pitfall.** `opcli artifacts fetch` without `--arch` detects
   the current machine's architecture and fetches only matching partials.  If a
   test VM runs on `amd64` but the test scenario targets `arm64` artifacts (e.g.
   a cross-arch test), use `--arch arm64` or `--arch all` explicitly.

2. **Cross-arch merge validation is deferred to publish time.** In the
   integration-test path each VM only merges its own arch's partials.  Validation
   checks that require all arches to be present simultaneously — such as verifying
   that every charm resource references a rock that was actually built — only run
   when `opcli artifacts fetch --arch all` is called at publish time.  A
   configuration error that is only visible in the cross-arch merge passes all
   integration tests and is caught only at release.

3. **Build job naming convention is a coupling point.** Early-bail failure
   detection in the fetch wait loops relies on matching GitHub job names against
   the pattern ``Build (charm|rock|snap) ...``.  This pattern is hard-coded in
   ``src/opcli/core/artifacts.py`` (``_BUILD_JOB_RE``) and must stay in sync with
   the ``name:`` field of the build job in ``build-artifacts.yml``.  If the
   workflow job name changes, the bail detection silently stops working and failed
   builds will only surface at timeout.



There could be several variations of this workflow depending on the process for the team/repository.

Following Platform Engineering practices, it will be similar to canonical/operator-workflows/.github/workflows/publish\_charm.yaml

- It will find the latest integration tests workflow run for the same commit tree hash (tree ID). That is, the same exact code.  
- From the integration test it will get the artifacts.build.yaml  
- Before uploading to CharmHub, it will inject a `version` file containing the first 8 characters of the publish workflow commit SHA into each fetched `.charm` archive for the same tested tree, unless the archive already contains a `version` file. This reusable-workflow-only behavior can be disabled with `inject-version: false`; local `opcli artifacts publish` never modifies charm archives.
- It will publish the artifacts to the desired channel in the correct order using `opcli artifacts publish --channel <channel>`. When different charms target different tracks, set `channel` per-charm in `artifacts.yaml` instead of (or in addition to) the global `--channel` flag.
- It will create one GitHub Release per published charm revision. The release body includes CharmHub publish metadata, then uses GitHub-generated release notes so the release includes sections such as "What's Changed" and "New Contributors".

The `opcli artifacts publish` command:

1. Reads `artifacts.build.yaml` to discover charms, their builds, and resource→rock mappings.
2. Reads `artifacts.yaml` (if present) to determine per-charm channel overrides.
3. Resolves the effective channel for each charm: per-charm `channel` in `artifacts.yaml` takes precedence over the global `--channel` flag. An error is raised if neither is set.
4. For each charm's OCI-image resources:
   - **Rock-backed resources** (`rock:` set): Uses the registry image ref from the build manifest (CI path) or the local `.rock` file (local path). Registry refs are passed with `docker://` transport.
   - **External resources** (no `rock:`): Reads `upstream-source` from the charm's `charmcraft.yaml` or `metadata.yaml`. Already-qualified transport strings (e.g. `docker://`, `oci-archive:`) are passed through unchanged.
5. Uploads each resource via `charmcraft upload-resource --image=<ref> --format=json`.
6. Uploads each `.charm` file via `charmcraft upload --release=<channel> --resource=<name>:<rev> --format=json`, binding resource revisions to the charm release.

For multi-base charms, each `.charm` file is uploaded separately (one per base) but shares the same resource revisions. For multi-arch rocks in CI, the registry ref is a manifest list covering all architectures.

## Design decisions

### **opcli**

 There are several alternatives for the implementation language:

- TypeScript/npm. Native option for GitHub Actions. However it introduces a language not widely used in the team, and for pinning using the Github “use:”,  it creates the requirement of having two repositories, one for the actions and one for the workflows. More risk of eventual platform lock-in.  
-  Python. The team is familiar with Python, although GitHub Actions integration is less native than TypeScript. Pinning to the current workflow commit SHA can be done with tools like canonical/get-workflow-version-action.   
- Go. Similar situation as with Python, but the team is not so familiar with Go.

The decision proposed for this specification is Python.

### **concierge**

Concierge is a purpose-built tool for charm test environments (Juju, LXD, MicroK8s, CK8s) in a single declarative file. The same  concierge.yaml  works locally and in CI — differences like mirrors are expressed as patches. It is maintained within Canonical with active development.

Its main limitation for the purpose of this specification is that there are functionalities not implemented that are required and will have to be done (see [https://github.com/canonical/concierge/issues/181](https://github.com/canonical/concierge/issues/181)) and repositories with unusual provisioning requirements fall outside of this tool. For those cases, the solution is to use spread.yaml prepare steps.

### **spread**

The current pipeline has no way to reproduce a test job locally. Spread is a Canonical developed tool, described as a “Convenient full-system test (task) distribution”, that among other things, allows running jobs (tests) in different environments.

Spread solves some of the problems in this specification:

- The tests and how they are executed are defined in spread configuration files.  
- With the help of two different backends, tests can be run in ci (in the current machine) or locally (in a lxd virtual machine).  
- Extra provisioning not supported by concierge can be added to spread configuration files. Spread prepare / prepare-each hooks act as very flexible extension points.

Besides that, with a use of a “virtual backend” that is not spread syntax, but expanded by the opcli tool:

- We can define the main provisioning and isolate the differences between running tasks in the GitHub and locally. Depending on how the virtual backend “integration-test” is expanded, it will provision and get/use the artifacts in a different way.

As disadvantages of using spread:

- It is a new tool that adds complexity. Using and debugging it requires familiarity with it.  
- Putting extra provisioning in spread configuration adds drift between repositories. The alternative considered was a bash file, which has similar issues.  
- For simple repositories, skipping the task.yaml file will simplify the repository, however spread does not allow it. Task.yaml is autogenerated by “opcli spread init”, so the cost is not high for the users.

Spread is gaining adoption in Canonical, and is already used in a similar way by tools like “charmcraft test” and “snapcraft test”.

Although spread adds complexity, the alternative was to create specific configuration files for jobs configuration, lifecycle hooks and extensions. Besides that, it is the tool that aligns local and CI testing with the backend expansion.

## Alignment of the proposed design with the design objectives

| Flexibility | Each phase (build, provision, test, publish) is an independent stage with file-based interfaces (artifacts.yaml → artifacts.build.yaml → spread). Stages can be used separately or composed. Support for monorepos, multi-arch builds, and combinatorial testing (Juju versions, substrates) through spread variants and concierge configurations. |
| :---- | :---- |
| Modularity | Concierge presets, spread prepare / prepare-each hooks, the CONCIERGE variant, and the stable artifacts.build.yaml interface allow users to extend behavior at well-defined points. External pipelines (publish, Trivy, load testing) can consume artifacts.build.yaml without coupling to the integration-test workflow. |
| Local testing | Every  opcli  command runs locally.  “opcli spread run“ and  “opcli provision prepare”  \+  “opcli pytest run” give two paths to run the full test cycle on a developer machine. |
| Performance | All current optimizations can be retained: matrix build of artifacts, matrix run of tests, provisioning running in parallel to building artifacts, rocks pushed to GHCR to avoid re-transfer, caching of rocks by git tree sha1. |
| Debuggability | Developers can reproduce any CI job locally with the same  opcli  commands. Intermediate files ( artifacts.yaml ,  artifacts.build.yaml ,  concierge.yaml ) are inspectable.  “opcli spread expand”  previews the full spread configuration. In GitHub, tmate can be used similarly as it is used now. |
| Convention over configuration | Standard layouts work with auto-generated files ( opcli artifacts init ,  opcli spread init ). Non-standard layouts override only the files that differ. |
| Reproducibility | In GitHub workflows, all references should be pinned. The opcli tool is installed from a pinned git ref (OPCLI\_GIT\_REF), concierge is pinned to a snap channel, and workflow references use commit SHAs. |
| Easy migration | A new integration\_test.yaml workflow can be created combining the new tooling similarly to the current one. With minimal repository changes (the only mandatory addition being concierge.yaml), migration can be done incrementally. This step can mostly be done automatically by AI. |
